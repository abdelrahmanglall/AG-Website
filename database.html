01. Introduction to Database Systems

Definition

A database is a collection of related data that represents some aspect of the real world.

يعني بيانات ليها علاقة ببعض ومش متخزنة عشوائي، بتعبّر عن نظام حقيقي زي جامعة أو شركة.

File System vs. Database System

Compared to traditional file systems, database systems offer significant advantages. While file systems often suffer from data redundancy, difficulty in modification, and weak security, database systems support shared data usage, fine-grained access control, and minimize duplication, ensuring data integrity and security.

File System Database System
Redundant data Shared data
Modification difficulties Controlled access rights
Weak security Reduced redundancy

Key database concepts include entities (real or logical objects such as Student, Course, or Employee), attributes (descriptions of entities such as Name, Age, Salary), records or tuples (complete data rows), fields (individual columns), and relationships (connections between two or more entities).

Database Development Stages

  1. Analysis: Understanding the problem and identifying the necessary data.
  2. Design: Creating an Entity Relationship Diagram (ERD) that models the data and then converting it into tables.
  3. Implementation: Executing the design using SQL commands.
  4. GUI: Developing user interfaces that interact with the database.

This structured approach ensures that databases are not only well-designed but also practical and user-friendly.

02. Database Environment & DBMS

Definition

A DBMS is a collection of programs that enables users to create and maintain a database.

هو البرنامج اللي بنتعامل من خلاله مع الداتابيز: نضيف، نعدّل، نحذف، ونأمّن البيانات.

Common Examples

  • MySQL
  • Oracle
  • SQL Server
  • MongoDB

Advantages

  • Data Sharing: Multiple users can access data concurrently.
  • Data Security: Authorization and authentication mechanisms protect sensitive information.
  • Data Integrity: Ensures accuracy and consistency of data over its lifecycle.
  • Backup and Recovery: Protects against data loss through systematic backups.
  • Concurrency Control: Manages simultaneous data access without conflicts.

These features make DBMS indispensable in maintaining robust, reliable, and secure data environments.

03. Analysis Phase

Definition

The analysis phase is the process of identifying the data requirements of the system.

يعني نفهم النظام محتاج بيانات إيه قبل أي تصميم.

Key Outputs

  • Identification of entities by name.
  • Quantification of entities.
  • Definition of attributes for each entity.
  • Establishment of basic relationships among entities.

By meticulously defining these elements, the analysis phase lays the groundwork for a coherent and efficient database design, preventing costly redesigns later.

المرحلة دي بتركّز على البيانات نفسها مش طريقة التخزين

04. ERD - Entity Relationship Diagram

The Entity Relationship Diagram (ERD) is a conceptual modeling tool that graphically represents the database structure. It provides a clear visualization of entities, their attributes, and the relationships between them.

Primary Components

Entity

An entity is a real-world object or concept that can be uniquely identified.

زي Student أو Employee.

Attribute

An attribute is a property that describes an entity.

زي الاسم أو السن.

Relationship

A relationship is an association among two or more entities.

يعني ارتباط بين كيانين.

Classifications

Types of Relationships

  • One-to-One (1:1)
  • One-to-Many (1:N)
  • Many-to-Many (M:N)

دي أشكال الربط بين الكيانات.

  • Entities: Strong Entity (Independent), Weak Entity (Dependant).
  • Attributes: Simple, Composite, Multivalued, Derived.
  • Relationship Degree: Unary, Binary, Ternary.

05. EERD - Enhanced ERD

The Enhanced Entity Relationship Diagram (EERD) extends the traditional ERD to handle more sophisticated and intricate database requirements. It introduces advanced concepts such as specialization, generalization, and inheritance.

Specialization

Specialization is the process of defining a set of subclasses of an entity type.

تقسيم كيان كبير لكيانات أصغر.

Generalization

Generalization is the process of defining a generalized entity from several entity types.

تجميع كيانات شبه بعض في كيان واحد.

Inheritance allows subclasses to acquire attributes and relationships from their superclass, promoting reuse and consistency.

06. Constraints, Specialization & Generalization

Disjointness Constraints

Specify whether an entity instance can simultaneously belong to multiple subclasses:

  • Disjoint: The entity belongs to only one subclass.
  • Overlapping: The entity may belong to multiple subclasses.

Completeness Constraints

Determine coverage of the superclass by subclasses:

  • Total Specialization: Every superclass entity must be a member of at least one subclass.
  • Partial Specialization: Some superclass entities might not belong to any subclass.

Hierarchies & Lattices

  • Hierarchy: Single inheritance tree.
  • Lattice: Multiple inheritance structure.
  • Category (Union Type): Subclasses formed by the union of multiple superclasses.

07. Mapping ERD to Relational Model

Relational Model

The relational model represents the database as a collection of relations (tables).

يعني نحول الرسمة لجداول

ERD Concept Relational Model
Entity Table
Attribute Column
Primary Key Primary Key (PK)

Relationship Mapping

  • One-to-One (1:1): Foreign key (FK) in one of the tables.
  • One-to-Many (1:N): FK in N-side table.
  • Many-to-Many (M:N): Separate joining table.

Special Considerations

  • Weak Entity: Mapped to a table with FK referencing owner entity plus partial key.
  • Multivalued Attribute: Requires a separate table.
  • Composite Attribute: Decomposed into simple attributes.
  • Derived Attribute: Not stored physically.
  • Recursive Relationship: Implemented via self-referencing FK.

08. SQL Fundamentals

Definition

SQL is a standard language for defining and manipulating relational databases.

هي اللغة اللي بنتعامل بيها مع الجداول.

  • DDL (Data Definition Language): CREATE, ALTER, DROP.
  • DML (Data Manipulation Language): INSERT, UPDATE, DELETE.
  • DQL (Data Query Language): SELECT.

Example: Create Table

CREATE TABLE Student (
    StudentID INT PRIMARY KEY,
    Name VARCHAR(100),
    Age INT
);

09. Advanced SQL Queries

Data Retrieval

Uses SELECT with clauses such as WHERE (conditions) and ORDER BY (sorting).

Aggregate Functions

COUNT, SUM, AVG, MIN, MAX.

Grouping

GROUP BY clusters rows sharing attribute values, often combined with HAVING to filter groups.

JOIN Operations

  • INNER JOIN: Returns matching rows.
  • LEFT JOIN: Returns all rows from the left table, matched or not.
  • RIGHT JOIN: Returns all rows from the right table.
  • FULL JOIN: Returns all rows from both tables.

Subqueries

Queries nested inside another query, allowing for complex, multi-step data retrieval logic.

10. NoSQL & MongoDB

Definition

NoSQL databases are databases that do not use the relational model.

بتستخدم مع البيانات الكبيرة وغير المنظمة.

Key Characteristics

  • Schema-less: Data can be stored without predefined schemas.
  • Scalable: Easily handles large volumes of data.
  • Distributed: Supports data distribution across multiple servers.

Common NoSQL Types

  • Key-Value Stores
  • Document Stores (e.g., MongoDB)
  • Column-Family Stores
  • Graph Databases

MongoDB

A prominent document-based NoSQL database that stores data in JSON-like BSON documents.

12. Practical Scenarios & Solutions

Q1: The Mapping Beast

Scenario: University Hospital

  1. Doctor (Superclass): Has SSN, Name, Specialty.
  2. Subclasses: 'Surgeon' (has operations_count) and 'Consultant' (has hourly_rate). Constraint: Disjoint, Total.
  3. Patient: Has ID, Name, Phone(Multi-valued).
  4. Treatment (Weak): Dependent on Patient. Has Name and Cost. (Partial Key: Name).
  5. Treats (Relation): M:N relationship between Doctor and Patient with attribute 'Date'.

ER Diagram

Step-by-Step Solution:

-- 1. Superclass (Doctor)
DOCTOR (SSN, Name, Specialty)

-- 2. Subclasses (Disjoint/Total -> Separate Tables with PK of Super)
SURGEON (Doc_SSN, Operations_Count)
CONSULTANT (Doc_SSN, Hourly_Rate)

-- 3. Strong Entity (Patient)
PATIENT (ID, Name)

-- 4. Multivalued Attribute (Phone) -> New Table
PATIENT_PHONES (Patient_ID, Phone_No)

-- 5. Weak Entity (Treatment) -> Owner PK + Partial Key
TREATMENT (Patient_ID, Treatment_Name, Cost)

-- 6. M:N Relation (Treats) -> Associative Table
TREATS (Doc_SSN, Patient_ID, Date)

Given Tables for SQL Challenges

Department: Dept_ID, Dept_Name (e.g., 1, CS)

Student: ID, Name, GPA, Dept_ID (e.g., 101, Ali, 3.5, 1)

Course: C_Code, Title, Credits (e.g., CS101, Java, 3)

1. Insert a new Student (Complex)

Insert a student named 'Mona', GPA 3.8, into the 'CS' Department (Warning: You only know the Department Name 'CS', not its ID!).

INSERT INTO Student (Name, GPA, Dept_ID)
VALUES ('Mona', 3.8, (
  SELECT Dept_ID FROM Department
  WHERE Dept_Name = 'CS'
));

2. Complex Select (Join + Condition)

Retrieve Names and GPA of students in 'CS' department who have a GPA higher than 3.0, ordered by GPA descending.

SELECT S.Name, S.GPA
FROM Student S
INNER JOIN Department D ON S.Dept_ID = D.Dept_ID
WHERE D.Dept_Name = 'CS'
AND S.GPA > 3.0
ORDER BY S.GPA DESC;

3. Aggregation (Grouping)

Find the Average GPA for each Department, but only show departments with more than 50 students. 

SELECT Dept_ID, AVG(GPA)
FROM Student
GROUP BY Dept_ID
HAVING COUNT(ID) > 50;
